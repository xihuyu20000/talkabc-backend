# TalkABC 后端服务 - 后端开发人员入门指南

## 一、项目概述

基于 Go + Gin + GORM + PostgreSQL + Redis 开发的社交应用后端 API 服务，支持 WebSocket 实时通信。

### 1.1 技术栈

| 技术 | 用途 | 说明 |
|------|------|------|
| Gin | HTTP Web框架 | 高性能路由和中间件，支持路由分组 |
| GORM | ORM数据库操作 | 简化数据库操作，自动迁移 |
| PostgreSQL | 关系型数据库 | 存储用户、消息、动态等数据 |
| Redis | 缓存/实时数据存储 | 存储验证码、在线状态、会话管理 |
| go-redis/v8 | Redis客户端 | Go语言Redis驱动 |
| gorilla/websocket | WebSocket库 | 实现实时双向通信 |
| JWT | 用户认证 | 无状态令牌认证 |
| Viper | 配置管理 | 读取YAML配置文件和命令行参数 |
| bcrypt | 密码加密 | 安全的密码哈希算法（cost=10） |
| zap | 结构化日志 | 高性能、结构化日志库 |
| lumberjack | 日志文件切割 | 自动按大小/时间分割日志文件 |
| snowflake | ID生成 | 分布式唯一ID生成器 |

## 二、开发环境搭建

### 2.1 环境要求

| 工具 | 版本 | 说明 |
|------|------|------|
| Go | >= 1.21 | 编程语言 |
| PostgreSQL | >= 14 | 关系型数据库 |
| Redis | >= 6.0 | 缓存数据库 |
| Git | >= 2.30 | 版本控制 |

### 2.2 安装步骤

```bash
# 1. 克隆项目
git clone <repository-url>
cd talkabc-backend

# 2. 安装依赖
go mod download

# 3. 创建数据库
createdb -U postgres talkabc

# 4. 配置环境
cp config/config.yaml config/config.dev.yaml
# 修改 config.dev.yaml 中的数据库密码等配置
```

### 2.3 启动服务

```bash
# 设置开发环境
export APP_ENV=dev

# 启动服务
go run ./cmd/server

# 或使用构建脚本
build_dev.bat
```

服务启动后会：
1. 加载配置文件（config.yaml）
2. 加载日志配置文件（config/logger.yaml）
3. 如果配置 `reset=1`，执行数据重置
4. 连接数据库（PostgreSQL，不存在则自动创建）
5. 自动创建所有数据表（根据models.go中的定义）
6. 初始化Redis连接
7. 初始化短信网关
8. 创建上传文件目录
9. 监听 8080 端口

### 2.5 Apifox API管理

项目使用 **Apifox** 作为API管理平台，后端开发人员需要在每次接口变更后更新Apifox配置文件。

**Apifox配置文件：** `talkabc.apifox.json`

**更新流程：**
1. 修改代码中的路由和handler
2. 更新 `talkabc.apifox.json` 文件中的对应接口定义
3. 导出Apifox集合并覆盖项目中的配置文件
4. 提交代码变更

**导入方式：**
1. 打开Apifox客户端
2. 点击"导入/导出" → "导入数据" → "从文件导入"
3. 选择 `talkabc.apifox.json` 文件
4. 点击"确定"完成导入

**后端开发常用功能：**

- **📋 API文档维护**：在Apifox中维护接口文档，确保与代码一致
- **🔄 接口调试**：开发完成后在Apifox中测试接口功能
- 🧪 **测试用例编写**：为每个接口编写测试用例，确保功能正确性
- 📊 **接口对比**：对比不同版本的API差异

**导出Apifox配置：**
1. 在Apifox中打开TalkABC项目
2. 点击"导入/导出" → "导出数据" → "导出为JSON"
3. 选择导出范围（全部或指定模块）
4. 保存为 `talkabc.apifox.json`，覆盖项目中的文件

## 四、架构说明

### 4.1 分层架构

项目采用经典的三层架构设计，每层有明确的职责：

```
┌─────────────────────────────────────────────────────────┐
│                    Handler 层                           │
│   (接收请求、参数验证、调用Service、返回响应)              │
├─────────────────────────────────────────────────────────┤
│                    Service 层                           │
│   (处理业务逻辑、数据转换、调用Repository)                │
├─────────────────────────────────────────────────────────┤
│                   Repository 层                         │
│   (直接与数据库交互、CRUD操作)                           │
├─────────────────────────────────────────────────────────┤
│                      数据库                              │
│               PostgreSQL + Redis                        │
└─────────────────────────────────────────────────────────┘
```

### 4.2 请求处理流程

```
客户端请求
    ↓
路由匹配 (router.go)
    ↓
请求日志中间件 (middleware/logger.go) - 生成request_id，记录请求开始
    ↓
CORS中间件 (cors) - 跨域处理
    ↓
JWT认证中间件 (middleware/jwt.go) - 私有接口验证
    ↓
Handler处理 (handler/*.go) - 参数验证、调用Service
    ↓
Service处理 (service/*.go) - 业务逻辑
    ↓
Repository处理 (repository/*.go) - 数据库操作
    ↓
数据库响应
    ↓
逐层返回
    ↓
请求日志中间件 - 记录请求结束（状态码、耗时）
    ↓
客户端收到JSON响应
```

### 4.3 WebSocket处理流程

```
客户端WebSocket连接请求
    ↓
WS Handler验证JWT token
    ↓
创建Client并注册到Hub
    ↓
启动ReadPump和WritePump协程
    ↓
接收客户端消息 → 解析 → 分发到对应处理函数
    ↓
处理业务逻辑 → 通过Hub推送给目标客户端
    ↓
客户端收到实时消息
```

## 五、配置说明

### 5.1 多环境配置

项目支持多环境配置，配置文件位于 `config/` 目录下：

| 文件 | 说明 | 是否上传Git |
|------|------|-------------|
| `config/config.yaml` | 默认配置（公共部分） | 是 |
| `config/config.dev.yaml` | 开发环境配置 | 否 |
| `config/config.test.yaml` | 测试环境配置 | 否 |
| `config/config.prod.yaml` | 生产环境配置（含环境变量引用） | 否 |

### 5.2 配置文件加载优先级

1. **APP_CONFIG 环境变量**：指定完整配置文件路径
2. **APP_ENV 环境变量**：根据环境选择配置文件（`config/config.{APP_ENV}.yaml`）
3. **默认配置文件**：`config/config.yaml`
4. **兼容旧版本**：`./config.yaml`

### 5.3 环境变量引用

生产环境配置支持 `${ENV_VAR}` 格式引用环境变量：

```yaml
database:
  host: "${DB_HOST}"
  password: "${DB_PASSWORD}"
```

### 5.4 各环境配置差异

| 配置项 | 开发环境 | 测试环境 | 生产环境 |
|--------|----------|----------|----------|
| 日志级别 | debug | info | warn |
| 日志格式 | console | json | json |
| 短信冷却 | 10秒 | 60秒 | 60秒 |
| CORS | `*` | 指定域名 | 指定域名 |
| JWT密钥 | dev_secret | test_secret | 环境变量 |
| 数据库 | talkabc | talkabc_test | 环境变量 |

### 5.6 配置文件示例

```yaml
# 系统配置
system:
  reset: 0                 # 重置标志(0=不重置, 1=重置所有数据)

# 日志配置（已独立到 config/logger.yaml，此处保留为兼容）

  level: debug          # 日志级别: debug, info, warn, error, fatal
  format: console       # 日志格式: console, json
  output: both          # 输出方式: console, file, both
  file_path: ./logs/app.log
  max_size: 100         # 单个日志文件最大大小(MB)
  max_backups: 30       # 保留的最大日志文件数
  max_age: 7            # 日志文件保留天数
  compress: true        # 是否压缩归档日志

# 安全配置
security:
  sms_valid_minutes: 5              # 短信验证码有效期（分钟）
  sms_cooldown_seconds: 60          # 短信发送冷却时间（秒）
  sms_hourly_limit: 10              # 每小时发送次数限制
  ip_register_hourly_limit: 10      # 注册时每小时每个IP发送次数限制
  ip_login_minute_limit: 10         # 登录时每分钟每个IP登录次数限制
  login_failure_lock_minutes: 5     # 登录失败锁定时间（分钟）

# 短信服务商配置
sms_provider:
  default: "aliyun"              # 默认短信服务商：aliyun, huawei, tencent
  aliyun:
    access_key_id: ""            # 阿里云AccessKey ID
    access_key_secret: ""        # 阿里云AccessKey Secret
    region_id: "cn-hangzhou"     # 阿里云区域ID
    sign_name: ""                # 短信签名
    template_code: ""            # 短信模板Code
  huawei:
    app_key: ""                  # 华为云AppKey
    app_secret: ""               # 华为云AppSecret
    sign_name: ""                # 短信签名
    template_id: ""              # 短信模板ID
  tencent:
    secret_id: ""                # 腾讯云SecretID
    secret_key: ""               # 腾讯云SecretKey
    region_id: "ap-guangzhou"    # 腾讯云区域ID
    sign_name: ""                # 短信签名
    template_id: ""              # 短信模板ID

# 服务器配置
server:
  port: 8080              # 服务监听端口

# 数据库配置
database:
  host: localhost         # 数据库服务器地址
  port: 5432             # PostgreSQL默认端口
  user: postgres          # 数据库用户名
  password: admin         # 数据库密码
  dbname: talkabc        # 数据库名称

  secret: your_secret_key # JWT签名密钥（生产环境应使用复杂密钥）
  expires_hour: 24        # Token过期时间（小时）

# 文件上传配置
upload:
  avatar_path: ./uploads/avatars    # 头像上传目录
  moment_path: ./uploads/moments     # 动态上传目录
  message_path: ./uploads/messages   # 消息文件上传目录

# Redis配置
```

### 7.2 日志模块

#### 核心设计

项目采用企业级日志方案，基于 **zap + lumberjack** 实现：

| 特性 | 说明 |
|------|------|
| 全局单例 | 整个项目只初始化一次 logger，禁止到处 new 日志对象 |
| 独立配置 | 通过 `config/logger.yaml` 统一控制日志行为 |
| 分层输出 | 同时输出到控制台和文件 |
| 自动切割 | 按大小/时间分割日志，自动清理过期日志 |
| 链路追踪 | 支持从 context 中提取 request_id/trace_id |
| 动态级别 | 运行时可通过 API 动态修改日志级别 |

#### 日志级别

支持以下日志级别（从低到高）：

| 级别 | 说明 | 适用场景 |
|------|------|----------|
| `debug` | 调试信息 | 开发环境，详细的调试日志 |
| `info` | 一般信息 | 默认级别，记录关键操作和状态 |
| `warn` | 警告信息 | 可能的问题，但不影响正常运行 |
| `error` | 错误信息 | 操作失败或异常情况 |
| `fatal` | 致命错误 | 严重错误，程序将退出 |

#### 使用方式

```go
import "backend/pkg/logger"

logger.Info("用户登录成功", "uid", uid)
logger.Warn("登录失败", "reason", "密码错误")
logger.Error("数据库连接失败", "error", err)
logger.Debug("调试信息", "data", data)
```

#### 请求日志中间件

系统自动记录每个HTTP请求的完整信息：

```
INFO  Request start - method: POST, path: /api/v1/auth/login/code, client_ip: 192.168.1.100
INFO  Request end - method: POST, path: /api/v1/auth/login/code, status: 200, latency: 15.3ms
```

#### 动态日志级别

支持通过 API 实时修改日志级别：

```bash
# 获取当前日志级别
curl -H "Authorization: Bearer xxx" http://localhost:8080/api/v1/system/log-level

# 修改日志级别为 debug
curl -H "Authorization: Bearer xxx" -X POST -d '{"level":"debug"}' http://localhost:8080/api/v1/system/log-level
```

#### 日志格式

**Console格式（开发环境）：**
```
[2026-07-03 10:30:45.123] [INFO] [service:talkabc] [env:development] [request_id:abc123] Request processed
```

**JSON格式（生产环境）：**
```json
{
  "time": "2026-07-03T10:30:45.123Z",
  "level": "info",
  "service": "talkabc",
  "env": "development",
  "request_id": "abc123",
  "message": "Request processed"
}
```

#### 敏感信息脱敏

系统提供工具函数对敏感信息进行脱敏处理：

| 函数 | 说明 | 示例 |
|------|------|------|
| `logger.MaskToken` | Token脱敏 | `abc123def456` → `abc1***456` |
| `logger.MaskSensitive` | 通用敏感信息脱敏 | 根据长度动态隐藏中间部分 |

#### PG日志存储

系统支持将ERROR/WARN级别日志写入PostgreSQL，便于关联业务数据排查问题：

**日志表结构：**

| 表名 | 用途 | 保留策略 |
|------|------|----------|
| `audit_logs` | 操作审计日志 | 90天 |
| `admin_change_logs` | 后台变更日志 | 永久（等保要求） |
| `compliance_logs` | 等保合规日志 | 永久（等保要求） |
| `exception_logs` | 异常日志（ERROR/WARN） | 30天 |

**使用方式：**

```go
import "backend/internal/service"

// 记录操作审计日志
service.RecordAuditLog(service.AuditLogParams{
    RequestID:    "abc123",
    UserID:       1,
    Operation:    "user_login",
    ResourceType: "user",
    ResourceID:   "1001",
    Action:       "create",
    Result:       1,
})

// 记录等保合规日志（登录成功）
service.RecordLoginSuccess("abc123", "trace123", 1, "uid1001", "138****1234", "192.168.1.1", "Mozilla/5.0")

// 记录等保合规日志（登录失败）
service.RecordLoginFailure("abc123", "trace123", "138****1234", "192.168.1.1", "Mozilla/5.0", "密码错误")

// 记录异常日志（自动关联订单、用户）
service.RecordError("abc123", "trace123", "支付失败", "payment_error", stackTrace, 1, "ORDER20240101001")

// 记录警告日志
service.RecordWarn("abc123", "trace123", "配置项缺失", "config_error", 1)

// 定期清理过期日志（建议每天凌晨执行）
service.CleanupOldLogs()
```

**等保合规日志类型：**

| 类型 | 说明 | 严重级别 |
|------|------|----------|
| `login_success` | 用户登录成功 | INFO |
| `login_failure` | 用户登录失败 | WARN |
| `logout` | 用户退出登录 | INFO |
| `privilege_escalation` | 权限提升 | CRITICAL |
| `data_access` | 数据访问 | INFO |
| `data_export` | 数据导出 | WARN |
| `configuration_change` | 配置变更 | WARN |
| `security_incident` | 安全事件 | CRITICAL |
| `access_denied` | 访问被拒绝 | WARN |

**日志哈希去重：**

异常日志通过SHA256哈希去重，相同的异常只记录一次，后续记录增加重试次数，避免日志爆炸。

**日志清理任务：**

```go
// 定期清理过期日志（建议每天凌晨执行）
service.CleanupOldLogs()
```

## 九、安全规则

### 9.1 密码安全

- 使用 bcrypt 加密存储（cost=10）
- 禁止明文存储密码
- 禁止返回密码信息
- 密码复杂度校验（≥8位，至少两种字符类型）

### 9.2 内容安全

昵称和签名需要通过安全检查：

```go
import "backend/pkg/security"

// 验证昵称
err := security.ValidateNickname(nickname)

// 验证签名
err := security.ValidateSignText(signText)
```

检查项：
- 长度限制
- 敏感词过滤
- URL过滤
- HTML过滤
- JavaScript过滤
- SQL注入过滤
- XSS过滤

### 9.3 认证安全

- JWT Token有效期2小时
- Refresh Token有效期7天
- 退出登录时清除所有Token
- 登录失败5次锁定15分钟
- IP登录频率限制（1分钟10次）

3. 生成 Swagger 文档到 `swagger/` 目录
4. 将 `swagger.json` 复制到项目根目录（方便 Apifox 导入）
5. 输出生成结果和文件路径

#### 16.2.2 导出选项

脚本支持以下参数：

```bash
# 指定输出目录
export_swagger.bat --output ./docs/api

# 仅检查不生成
export_swagger.bat --check

# 详细输出
export_swagger.bat --verbose
```

### 16.3 导入到Apifox

#### 16.3.1 方式一：从文件导入（推荐）

1. 打开 Apifox 客户端
2. 点击左侧菜单的 **API文档**
3. 点击顶部的 **导入** 按钮
4. 选择 **从文件导入**
5. 选择项目中的 `swagger/swagger.json` 文件
6. 点击 **确定** 完成导入

#### 16.3.2 方式二：从URL导入（支持自动同步）

1. 确保后端服务已启动
2. 在 Apifox 中点击 **导入** → **从URL导入**
3. 输入：`http://localhost:8080/swagger/doc.json`
4. 勾选 **定时同步**（可选，建议开启）
5. 点击 **确定** 完成导入

#### 16.3.3 方式三：导入Apifox配置文件

项目根目录的 `talkabc.apifox.json` 包含完整的 Apifox 配置：

1. 在 Apifox 中点击 **导入/导出** → **从文件导入**
2. 选择 `talkabc.apifox.json` 文件
3. 点击 **确定** 完成导入

### 16.4 Apifox环境配置

导入后需要配置环境变量：

1. 在 **环境管理** 中添加环境
2. 设置 **基础URL**：`http://localhost:8080/api/v1`
3. 添加 **Authorization** 头：
   - 键：`Authorization`
   - 值：`Bearer {{token}}`

### 16.5 工作流建议

```
开发新接口 → 添加Swagger注释 → 运行导出脚本 → 导入Apifox → 测试验证
```

每次接口变更后：
1. 更新 Handler 层的 Swagger 注释
2. 运行 `export_swagger.bat`（或 `./export_swagger.sh`）
3. 在 Apifox 中重新导入或同步文档
4. 测试接口功能

### 16.6 常见问题

**Q: 导入后部分接口没有显示？**

确保 Handler 函数有完整的 Swagger 注释，特别是 `@Router` 标签。

**Q: 如何让 Swagger 支持 JWT 认证？**

在注释中添加 `@Security BearerAuth`，Apifox 会自动识别。

**Q: 定时同步功能有什么用？**

启用后，Apifox 会定期从 `http://localhost:8080/swagger/doc.json` 获取最新文档，保持与代码同步。
