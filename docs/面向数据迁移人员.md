# TalkABC 后端服务 - 数据迁移人员入门指南

## 一、数据迁移概述

### 1.1 迁移场景

| 场景 | 说明 |
|------|------|
| 系统上线 | 从旧系统迁移数据到新系统 |
| 数据库升级 | PostgreSQL版本升级 |
| 数据备份恢复 | 从备份恢复数据 |
| 数据同步 | 多环境数据同步（开发→测试→生产） |
| 数据清理 | 清理无效数据、测试数据 |

### 1.2 迁移原则

```
1. 最小化停机时间
2. 确保数据完整性
3. 保持数据一致性
4. 可回滚操作
5. 充分测试验证
```

### 1.3 迁移流程

```
Step 1: 数据调研与分析
Step 2: 制定迁移方案
Step 3: 数据备份
Step 4: 数据迁移
Step 5: 数据验证
Step 6: 上线切换
Step 7: 后续监控
```

2. 打开Apifox客户端，点击"导入/导出" → "导入数据" → "从文件导入"

**数据迁移人员常用功能：**

- 📋 **API文档查阅**：查看接口的请求参数和响应数据结构
- 📊 **数据模型分析**：了解各表之间的关联关系和数据流转
- 🧪 **接口测试**：验证数据迁移后接口功能是否正常

**数据结构查看：**
1. 在Apifox中选择某个接口
2. 查看"请求参数"和"响应示例"

```
验证流程：
```

## 三、数据迁移方案

### 3.1 方案一：SQL文件迁移

#### 导出数据

```bash
# 导出所有数据
pg_dump -h localhost -U postgres talkabc > talkabc_full_backup.sql

# 导出指定表
pg_dump -h localhost -U postgres talkabc -t users -t user_friend > talkabc_users.sql

# 导出数据（不包含表结构）
pg_dump -h localhost -U postgres talkabc -a > talkabc_data_only.sql

# 导出表结构（不包含数据）
pg_dump -h localhost -U postgres talkabc -s > talkabc_schema_only.sql
```

#### 导入数据

```bash
# 创建目标数据库
createdb -U postgres talkabc_new

# 导入完整数据
psql -h localhost -U postgres talkabc_new < talkabc_full_backup.sql

# 导入指定表
psql -h localhost -U postgres talkabc_new < talkabc_users.sql

# 导入数据（不包含表结构）
psql -h localhost -U postgres talkabc_new < talkabc_data_only.sql
```

### 3.2 方案二：数据同步工具

#### 使用 pg_dump 和 pg_restore

```bash
# 导出为压缩格式
pg_dump -h localhost -U postgres talkabc | gzip > talkabc_backup.sql.gz

# 导入压缩文件
gunzip -c talkabc_backup.sql.gz | psql -h localhost -U postgres talkabc_new

# 使用 pg_restore
pg_dump -h localhost -U postgres talkabc -Fc > talkabc_backup.dump
pg_restore -h localhost -U postgres -d talkabc_new talkabc_backup.dump
```

#### 使用 COPY 命令

```bash
# 导出CSV
psql -h localhost -U postgres talkabc -c "COPY users TO '/tmp/users.csv' WITH CSV HEADER;"

# 导入CSV
psql -h localhost -U postgres talkabc_new -c "COPY users FROM '/tmp/users.csv' WITH CSV HEADER;"
```

### 3.3 方案三：自定义迁移脚本

#### 创建迁移脚本

```bash
cat > migrate_users.sh << 'EOF'
#!/bin/bash

# 源数据库
SOURCE_DB="talkabc_old"
SOURCE_HOST="old-server"

# 目标数据库
TARGET_DB="talkabc"
TARGET_HOST="new-server"

# 导出用户数据
pg_dump -h $SOURCE_HOST -U postgres $SOURCE_DB -t users -a > users_data.sql

# 转换数据格式（如需）
# sed -i 's/old_value/new_value/g' users_data.sql

# 导入数据
psql -h $TARGET_HOST -U postgres $TARGET_DB < users_data.sql

echo "用户数据迁移完成"
EOF
```

#### 执行迁移脚本

```bash
chmod +x migrate_users.sh
./migrate_users.sh
```

## 四、数据迁移步骤

### 4.1 准备阶段

#### Step 1: 数据调研

```bash
# 统计各表数据量
psql -h localhost -U postgres talkabc -c "
SELECT 
    relname AS table_name,
    reltuples AS row_count
FROM pg_class 
WHERE relkind = 'r' 
ORDER BY reltuples DESC;
"

# 检查数据完整性
psql -h localhost -U postgres talkabc -c "
SELECT 
    table_name,
    column_name,
    is_nullable
FROM information_schema.columns 
WHERE table_name IN ('users', 'chat_messages');
"

# 检查索引状态
psql -h localhost -U postgres talkabc -c "
SELECT 
    indexrelname AS index_name,
    relname AS table_name,
    idx_scan AS scan_count
FROM pg_stat_user_indexes 
ORDER BY idx_scan DESC;
"
```

#### Step 2: 制定迁移方案

```
迁移方案模板：
==============

1. 迁移范围：
   - users 表：全量迁移
   - chat_messages 表：最近3个月数据
   - user_moments 表：全量迁移

2. 迁移顺序：
   Step 1: 基础数据（hobby_tag, gift）
   Step 2: 用户数据（users）
   Step 3: 关系数据（user_friend, user_focus）
   Step 4: 消息数据（chat_messages）
   Step 5: 动态数据（user_moments）

3. 迁移时间：
   - 预计停机时间：30分钟
   - 开始时间：凌晨2点

4. 回滚方案：
   - 迁移前备份数据库
   - 如果迁移失败，从备份恢复
```

#### Step 3: 数据备份

```bash
# 全量备份
pg_dump -h localhost -U postgres talkabc > talkabc_migration_backup.sql

# 备份到远程服务器
pg_dump -h localhost -U postgres talkabc | ssh user@backup-server "cat > /backup/talkabc_migration_backup.sql"

# 压缩备份
pg_dump -h localhost -U postgres talkabc | gzip > talkabc_migration_backup.sql.gz
```

### 4.2 迁移阶段

#### Step 4: 创建目标数据库

```bash
# 创建数据库

# 创建用户并授权
psql -h localhost -U postgres -c "

GRANT ALL PRIVILEGES ON DATABASE talkabc_new TO talkabc;
"
```

#### Step 5: 初始化表结构

```bash
# 方案一：使用GORM自动迁移
go run ./cmd/server -migrate

# 方案二：使用SQL文件
psql -h localhost -U postgres talkabc_new < schema.sql

# 方案三：使用pg_dump导出表结构
pg_dump -h localhost -U postgres talkabc_old -s | psql -h localhost -U postgres talkabc_new
```

#### Step 6: 执行数据迁移

```bash
# 迁移用户数据
pg_dump -h old-server -U postgres talkabc_old -t users -a | psql -h new-server -U postgres talkabc_new

# 迁移消息数据（最近3个月）
pg_dump -h old-server -U postgres talkabc_old -t chat_messages -a --where="send_time > '2026-04-01'" | psql -h new-server -U postgres talkabc_new

# 迁移动态数据
pg_dump -h old-server -U postgres talkabc_old -t user_moments -a | psql -h new-server -U postgres talkabc_new

# 迁移关系数据
pg_dump -h old-server -U postgres talkabc_old -t user_friend -a | psql -h new-server -U postgres talkabc_new
pg_dump -h old-server -U postgres talkabc_old -t user_focus -a | psql -h new-server -U postgres talkabc_new
```

## 五、数据迁移注意事项

### 5.1 数据格式转换

#### 密码字段转换

```bash
# 旧系统密码可能使用不同的加密方式
# 需要转换为bcrypt加密

# 示例：从MD5转换为bcrypt
# 旧数据：password = MD5('123456')
# 新数据：password = bcrypt.hashpw('123456', bcrypt.gensalt())
```

#### 日期格式转换

```bash
# 确保日期格式正确
psql -h new-server -U postgres talkabc_new -c "
ALTER TABLE users 
ALTER COLUMN last_seen_at TYPE timestamp 
USING last_seen_at::timestamp;
"
```

#### JSON字段转换

```bash
# 确保JSON字段格式正确

ALTER TABLE users
```

### 5.2 数据增量同步

#### 方案一：触发器同步

```sql
-- 在源数据库创建触发器
CREATE TRIGGER sync_users_after_insert
AFTER INSERT ON users
FOR EACH ROW
EXECUTE FUNCTION sync_to_new_db();

-- 在目标数据库创建同步函数
CREATE OR REPLACE FUNCTION sync_to_new_db()
RETURNS TRIGGER AS $$
BEGIN
    -- 插入到目标数据库
    INSERT INTO new_db.users (...)
    VALUES (...);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

#### 方案二：定时同步

```bash
# 创建定时同步脚本
cat > sync_data.sh << 'EOF'
#!/bin/bash

# 同步新增用户（基于created_at）
pg_dump -h old-server -U postgres talkabc_old -t users -a --where="created_at > '$(date -d '1 hour ago' '+%Y-%m-%d %H:%M:%S')'" | psql -h new-server -U postgres talkabc_new

# 同步新增消息
pg_dump -h old-server -U postgres talkabc_old -t chat_messages -a --where="send_time > '$(date -d '1 hour ago' '+%Y-%m-%d %H:%M:%S')'" | psql -h new-server -U postgres talkabc_new

echo "数据同步完成"
EOF

# 添加到crontab（每小时执行一次）
# 0 * * * * /opt/talkabc/scripts/sync_data.sh
```

### 5.4 性能优化

#### 分批迁移

```bash
# 分批迁移大数据表
for i in $(seq 0 1000 10000); do
    pg_dump -h old-server -U postgres talkabc_old -t chat_messages -a --where="id BETWEEN $i AND $((i+999))" | psql -h new-server -U postgres talkabc_new
    echo "迁移完成: $i - $((i+999))"
done
```

#### 禁用索引和约束

```bash
# 迁移前禁用索引和约束
psql -h new-server -U postgres talkabc_new -c "
ALTER TABLE users DISABLE TRIGGER ALL;
ALTER TABLE chat_messages DISABLE TRIGGER ALL;
"

# 迁移后启用索引和约束
psql -h new-server -U postgres talkabc_new -c "
ALTER TABLE users ENABLE TRIGGER ALL;
ALTER TABLE chat_messages ENABLE TRIGGER ALL;
"
```

#### 并行迁移

```bash
# 并行迁移多个表
(pg_dump -h old-server -U postgres talkabc_old -t users -a | psql -h new-server -U postgres talkabc_new) &
(pg_dump -h old-server -U postgres talkabc_old -t user_friend -a | psql -h new-server -U postgres talkabc_new) &
(pg_dump -h old-server -U postgres talkabc_old -t user_focus -a | psql -h new-server -U postgres talkabc_new) &
wait
echo "所有表迁移完成"
```

## 六、数据回滚

### 6.1 回滚方案

## 九、常见问题

### Q1: 迁移过程中数据不一致？
A: 使用分批迁移和增量同步方案，迁移前锁定源数据库写入

### Q2: 大数据表迁移太慢？
A: 使用并行迁移、禁用索引约束、分批处理

### Q3: 密码加密方式不同？
A: 在迁移脚本中添加密码转换逻辑，确保转换为bcrypt加密

### Q4: JSON字段格式错误？
A: 使用 `USING aim::json` 进行类型转换，验证JSON格式

### Q5: 迁移后服务无法启动？
A: 检查数据库连接配置、表结构是否完整、索引是否存在

## 十、紧急处理流程

### 10.1 迁移失败

```
1. 停止迁移进程
2. 记录错误日志
3. 分析失败原因
4. 从备份恢复数据
5. 修正迁移方案
6. 重新执行迁移
```

### 10.2 数据不一致

```
1. 停止新系统写入
2. 对比源数据库和目标数据库
3. 找出不一致的数据
4. 手动修正或重新迁移
5. 验证数据一致性
6. 恢复新系统写入
```

### 10.3 服务异常

```
1. 停止新系统服务
2. 切换回原系统
3. 分析服务异常原因
4. 修复问题
5. 重新迁移或切换
```
